# DentureCare 설계 개요 및 기술 스택

> 작성일: 2026-07-13 | 기준: 프로토타입 v12 분석
> 이 문서는 Phase 1(설계)의 기준 문서입니다. 모든 개발은 이 설계를 따릅니다.

## 1. 프로토타입 v12 분석 결과

프로토타입에는 4가지 사용자 유형(액터)이 구현되어 있다:

| 액터 | 대상 | 주요 화면 |
|---|---|---|
| A1 | 틀니 사용자 본인 (65세+) | 홈(오늘 할 일), 행동 실행, 진행률, 치과 검진, 설정 |
| A2 | 가족 보호자 (자녀·배우자) | 홈(부모님 현황), 리포트, 알림 센터, 설정 |
| A3 | 요양보호사·시설 관리자 | 어르신 목록, 상세, 시설 통계, 인수인계, 설정 |
| A5 | 시스템 관리자 | 대시보드, 시설 관리, 사용자 관리, 설정 |

핵심 도메인 로직 (프로토타입에서 검증됨 — 그대로 계승):
- **일일 루틴 5회**: 기상(07:00), 아침 식후(08:00), 점심 식후(12:30), 저녁 식후(19:00), 취침 전(22:30)
- **리콜 주기 자동 계산**: 틀니 제작 시점 기준 4단계 (적응기 1개월 → 1년차 3개월 → 안정기 6개월 → 5년+ 6개월/교체 검토), Tallgren(1972) 골흡수 근거
- **습관화 지표**: streak(연속 일수) + automaticity(자동화 점수), phase 1~3 (30일/90일 기준)
- **응급 신고**: 잇몸 통증, 틀니 부적합 등 → 보호자에게 알림
- **접근성**: 글자 크기 조절 모드(normal/large)

## 2. MVP 범위 결정

**MVP = A1 + A2만.** A3(시설)와 A5(관리자)는 프로토타입 화면은 있지만 정식 개발에서는 2차 버전으로 미룬다.

근거:
- 차별화 핵심은 "틀니 특화 루틴 + 가족 연동" — A1/A2만으로 성립
- A3는 시설 계약(B2B)이 선행되어야 의미 있음 → 수익화 단계에서 재검토
- A5는 사용자가 쌓인 후에 필요 → Supabase 대시보드로 당분간 대체 가능
- 1인 개발로 4개 액터 동시 개발은 출시 지연 위험

## 3. 기술 스택 (확정)

| 계층 | 선택 | 이유 |
|---|---|---|
| 프론트엔드 | React + Vite + TypeScript | 프로토타입이 이미 React 문법. Vite는 가장 표준적인 빌드 도구 |
| 스타일 | 프로토타입의 인라인 스타일 → CSS 변수 + 모듈로 정리 | 디자인 토큰(colors)이 이미 정의돼 있음 |
| 백엔드 | Supabase (Postgres + Auth + Realtime) | 무료 티어, 로그인 내장, 실시간 구독으로 보호자 연동 구현 용이 |
| 알림 | Capacitor local-notifications (기기 알림) | 루틴 시간 알림은 서버 불필요. 보호자 응급 알림은 Supabase Realtime |
| 앱 패키징 | Capacitor 6 (기존 설정 유지) | 이미 설정 완료 |
| 배포 | Vercel (웹 PWA 우선) | 깃허브 연동 자동 배포 |
